**问题：** 两种格式完全不同，需要统一。

## 六、发现的问题

### 问题1：维度 vs 指标 判定歧义（中等优先级）

**描述：** 当量表只有总分、无数个子维度时（如 PHQ-9），total_score 放在 dimensions 还是 metrics？
**建议：** 明确判定规则：

- 如果量表有多个维度/因子，每个都有独立解释 → 各维度放在 dimensions
- 如果量表只有总分一个指标 → 放在 metrics
- 或者：总分永远放在 metrics，子维度放在 dimensions

### 问题2：screening 格式不统一（高优先级）

**描述：** 方案1和方案2生成的 screening 字段结构完全不同。
**建议：** 统一 JSON 结构定义，确保两种方案输出格式一致。

### 问题3：解释文字长度差异（低优先级）

**描述：** 方案2的解释文字普遍更长。
**建议：** 在 prompt 中明确解释文字的字数要求（如 30-50 字）。

## 七、结论与建议

### ✅ 优点（一步合并方案）

1. **Token 节省 76-77%**，大幅降低 API 调用成本
2. **JSON 结构基本一致**，维度、指标、解释条目数量相同
3. **公式类型完全一致**，所有维度的 SUM/AVG 判定相同
4. **解释文字更详细**，信息损失更少

### ⚠️ 需要改进

1. **统一 dimensions/metrics 判定规则**
2. **统一 screening JSON 结构**
3. **控制解释文字长度**

### 📋 建议

1. **可先用一步合并方案**，在 SSRS 这类多维度量表上效果良好
2. **PHQ-9 类单指标量表**，建议进一步验证维度/指标归属是否影响计分引擎
3. **统一 JSON 结构定义**，确保两种方案输出完全兼容
