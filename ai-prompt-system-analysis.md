## 四、方案设计

### 方案 A：集中式 Markdown 管理（推荐 ⭐）

**核心思路**：所有提示词源文件统一在 `prompts/` 目录，构建管道统一输出。

```
prompts/
  ├── prompts.yaml           ← 新增：全局配置（谁是最新的、依赖关系）
  ├── CSV导入.md
  ├── JSON导入.md
  ├── 计分提取.md
  ├── meta-prompt.md
  ├── 输出格式.md             ← 新增：从 index.html 抽离
  ├── NPC过渡语.md            ← 新增：NPC 过渡语模板
  └── 系统模板.md             ← 新增：TEMPLATES 文本
        │
        ▼ build-prompts.py（不需要改动）
        │
        ▼ default-prompts.js（自动包含全部）
```

| 维度   | 评价                                                                           |
| ------ | ------------------------------------------------------------------------------ |
| 优点   | ✅ 单一来源，改一处全生效 ✅ 版本管理一致 ✅ 现有体系无缝衔接 ✅ P0 已打通     |
| 缺点   | ⚠️ NPC 过渡语是运行时动态数据，不适合静态文件 ⚠️ localStorage 版本同步仍需人工 |
| 风险   | 🟢 低（与现有流程一致）                                                        |
| 改动量 | 新增 3 个 `.md` 文件 + 删除 2 处硬编码                                         |

### 方案 B：混合式（源文件 + 数据库）

**核心思路**：静态提示词存 `prompts/`，动态提示词（NPC 过渡语、量表级 System Prompt）存 CloudBase 数据库。

```
prompts/           ← 静态提示词（构建时发布）
CloudBase          ← 动态提示词（运行时读写）
  ├── config/npc_prompts  ← NPC 过渡语池
  └── scales/{id}.aiDiag  ← 量表级 System Prompt
```

| 维度 | 评价                                                            |
| ---- | --------------------------------------------------------------- |
| 优点 | ✅ 动态数据实时同步 ✅ NPC 过渡语可多端共享 ✅ 有数据库级回滚   |
| 缺点 | ⚠️ 需要新增 API 端点 ⚠️ 离线场景需要 fallback ⚠️ 实现复杂度较高 |
| 风险 | 🟡 中（需改前端+后端）                                          |
| 适用 | 团队协作、多管理员同时操作                                      |

### 方案 C：最小改动方案

**核心思路**：只修复 P1-P4，不做架构调整。

```
改动清单:
  ① 修复 TEMPLATES 读取自 default-prompts.js 而非硬编码
  ② 增加 SP_DATA_VERSION 自动递增（构建时注入时间戳）
  ③ 在 build-prompts.py 中增加版本一致性检查
```

| 维度 | 评价                                      |
| ---- | ----------------------------------------- |
| 优点 | ✅ 改动最小 ✅ 零风险 ✅ 今天可完成       |
| 缺点 | ⚠️ 未解决根本架构问题 ⚠️ 输出格式仍硬编码 |
| 风险 | 🟢 极低                                   |

## 八、测试验证机制

### 8.1 测试分类

| 层级           | 测试方法                                        | 触发时机 | 工具                                |
| -------------- | ----------------------------------------------- | -------- | ----------------------------------- |
| **语法检查**   | JS 语法校验 + JSON parse 完整性                 | 每次构建 | `node -c` + `python3 -c json.loads` |
| **变量完整性** | 检查必需变量 `{scaleName}` `{score}` 等是否存在 | 每次构建 | 正则匹配                            |
| **指令一致性** | 检测矛盾指令（如「写"无"」vs「省略」）          | 每次构建 | 关键词模式库                        |
| **人工抽检**   | 用 1-2 个已知量表跑一遍完整流程                 | 发布前   | 后台手动操作                        |
| **A/B 对比**   | 新旧提示词输出对比，评分一致性                  | 灰度期   | 人工或简单脚本                      |

### 8.2 构建时自动检查（集成到 build-prompts.py）

````python
# 构建后执行以下检查：
def validate_prompt(prompt):
    errors = []

    # 1. 变量完整性：System Prompt 必须含 5 个变量
    if prompt.id == 'meta':
        for var in ['{scaleName}', '{score}', '{level}', '{dimensions}', '{answers}']:
            if var not in prompt.content:
                errors.append(f"缺少必需变量: {var}")

    # 2. 矛盾指令检测
    contradictions = [
        ("写'无'", "省略"),
        ("输出筛查", "不标筛查"),
        ("必须输出", "可选"),
    ]
    for a, b in contradictions:
        if a in prompt.content and b in prompt.content:
            errors.append(f"⚠️ 检测到可能矛盾的指令: '{a}' vs '{b}'")

    # 3. JSON 示例有效性
    json_blocks = re.findall(r'```json\n(.*?)```', prompt.content, re.DOTALL)
    for i, block in enumerate(json_blocks):
        try:
            json.loads(block)
        except:
            errors.append(f"JSON 示例 #{i+1} 无效")

    return errors
````

```
│  L1: Git 回滚（最可靠）    耗时: 1min    │
│  git revert → build-prompts → deploy    │
├──────────────────────────────────────────┤
│  L2: 版本切换（管理后台）  耗时: 30s     │
│  后台「系统提示词」Tab 切换版本下拉框    │
├──────────────────────────────────────────┤
│  L3: 本地备份（应急）      耗时: 2min    │
│  cp prompts/*.md prompts/backup/         │
├──────────────────────────────────────────┤
│  L4: localStorage 清空    耗时: 10s      │
│  控制台: localStorage.removeItem('psy_sys_prompts') │
│  → 下次刷新自动从 default-prompts.js 加载 │
└──────────────────────────────────────────┘
```

### 9.2 回滚触发条件

| 条件                   | 回滚层级  | 决策人       |
| ---------------------- | --------- | ------------ |
| 构建失败（语法错误）   | L1        | 自动阻断部署 |
| 构建成功但 AI 输出异常 | L2        | 管理员判断   |
| 严重影响线上用户       | L1 + 通知 | 立即执行     |
| localStorage 缓存混乱  | L4        | 管理员判断   |

### 9.3 构建时的自动备份

`build-prompts.py` 在构建前自动执行：

当前系统**无**多用户/权限系统。只有一个管理员 `oyORU3XImvO_rYAWBUTMNm89-3v0`。

### 10.2 建议权限分级

| 角色           | 能做什么                         | 不能做什么       |
| -------------- | -------------------------------- | ---------------- |

### 10.3 审批流（轻量级，适合当前小团队）

```
提示词修改
    │
    ├─ 语法检查通过？
    │   ├─ 否 → 阻断
    │   └─ 是 → 继续
    │
    ├─ 改动范围判断

```

**不设正式审批流程**——当前仅你一人管理，过度流程化是负担。改为**自动检查 + 人工抽检**。

## 十四、SP_DATA_VERSION 策略

### 14.1 当前问题

```javascript
// admin-legacy.html:14572
const SP_DATA_VERSION = 27; // 手动维护！
```

每次改提示词忘记 +1，localStorage 永远不会更新。

### 14.2 建议方案：构建时自动注入

```python
# build-prompts.py 构建时计算
import hashlib, json

all_content = json.dumps(prompts, sort_keys=True, ensure_ascii=False)
md5 = hashlib.md5(all_content.encode()).hexdigest()[:8]

# 写入 default-prompts.js
js_content = f'window.DEFAULT_PROMPTS = {json.dumps(prompts)};\n'
js_content += f'window.PROMPTS_VERSION = "{md5}";\n'

# admin-legacy.html 读取
// const SP_DATA_VERSION = window.PROMPTS_VERSION || 'manual';
// 不再手动维护！
```

### 14.3 对比

| 方案       | 优点                     | 缺点         |
| ---------- | ------------------------ | ------------ |
| 手动递增   | 简单                     | 🔴 容易忘记  |
| 时间戳     | 每次不同                 | 无意义的变化 |
| **MD5** ⭐ | 内容驱动，相同内容同版本 | 需构建时计算 |

## 十五、命名规范

### 15.1 文件命名

**规则：中文语义优先，英文作为补充说明，连字符分隔**

```
✅ 计分提取.md              ← 一眼看懂
✅ 计分提取-prompt.md        ← 明确是提示词
✅ JSON导入.md               ← 中文+英文混合
✅ CSV导入.md                ← 同上
✅ meta-prompt.md            ← 纯英文（仅 Meta 这类约定俗成的词）
✅ NPC过渡语.md              ← 中文优先

❌ csv-import.md             ← 英文不直观
❌ scoring-extract.md        ← 同上
❌ CSV导入文件生成提示词.md   ← 太长
❌ 计分规则提取提示词_v3.md  ← 不要版本号在文件名里
```

**命名优先级**：

| 优先级 | 格式          | 适用场景               | 示例                         |
| :----: | ------------- | ---------------------- | ---------------------------- |
|   1    | **纯中文**    | 功能明确，有共识中文名 | `计分提取.md`、`量表导入.md` |
|   2    | **中文-英文** | 需要补充技术含义       | `JSON导入.md`、`CSV导出.md`  |
|   3    | **纯英文**    | 技术术语无合适中文     | `meta-prompt.md`             |

**不需要的内容**：

- ❌ 类型前缀：不用 `source-`、`template-`（目录即分类）
- ❌ 版本号：不用 `v3.1`、`2026-05`（放 YAML frontmatter 里）
- ❌ 作者名：不用 `rich-计分提取.md`

### 15.2 内部版本分隔格式

**规则：`## ✅ v{major}.{minor} — {一句话说明}`**

```
✅ 格式示例：

## ✅ v1.0 — 初始版本
（提示词正文）

## ✅ v2.0 — 新增 allowsRepeat 题型计分规则
（提示词正文）

## ✅ v3.1 — 修复筛查条件判断逻辑，新增 parent-child 题型
（提示词正文）


❌ 错误格式：

## v1                          # 缺少 ✅ 标记
## version-2.0 — 新增功能      # 多了 version- 前缀
### ✅ v2.0 — 修复              # 用了三级标题
## 📦 v1.0 — 旧版               # 📦 仅用于旧版本标记
```

**版本标记含义**：

| 标记 | 含义     | 何时使用                 |
| ---- | -------- | ------------------------ |
| `✅` | 活跃可用 | 当前仍可用的版本         |
| `📦` | 已归档   | 不再推荐使用，保留供参考 |

**版本号规则**：

| 规则       | 说明                        | 示例                                            |
| ---------- | --------------------------- | ----------------------------------------------- |
| 数字格式   | `major.minor`，纯数字无前缀 | `1.0`, `2.0`, `3.1`                             |
| 主版本递增 | 重大结构变化                | `1.0 → 2.0`                                     |
| 次版本递增 | 小修/补丁                   | `3.0 → 3.1`                                     |
| 不做时间戳 | 版本号不反映日期            | `2026-05-05` 在 YAML frontmatter 的 `date` 字段 |

### 15.3 YAML Frontmatter 规范：id vs name 职责分离

```yaml
# 文件: 计分提取.md
---
id: scoring-extract # ← 英文 kebab-case，代码引用用
name: '计分规则提取提示词' # ← 中文，后台 UI 显示用
icon: 🎯
note: '用于 AI 从手册提取计分规则'
currentVersion: '3.1'
---
```

| 字段   |        语言         | 用途                                                 | 示例                           |
| ------ | :-----------------: | ---------------------------------------------------- | ------------------------------ |
| 文件名 |    **中文优先**     | Git、文件管理器浏览                                  | `计分提取.md`                  |
| `id`   | **英文 kebab-case** | 代码 `prompts.find(p => p.id === 'scoring-extract')` | `scoring-extract`              |
| `name` |      **中文**       | 后台 UI 下拉框、表格显示                             | `"计分规则提取提示词"`         |
| `note` |      **中文**       | 一句话说明                                           | `"用于 AI 从手册提取计分规则"` |

**原则**：文件名给人看（中文），`id` 给代码用（英文），`name` 给 UI 用（中文）。

### 15.4 现有文件重命名映射

| 当前文件名                  | 重命名为      | 原因           |
| --------------------------- | ------------- | -------------- |
| `CSV导入文件生成提示词.md`  | `CSV导入.md`  | 中文优先，简洁 |
| `JSON导入文件生成提示词.md` | `JSON导入.md` | 同上           |
| `计分规则提取提示词.md`     | `计分提取.md` | 同上           |
| `meta-prompt.md`            | 不变 ✅       | 纯英文约定俗成 |

### 15.5 提示词内容中的变量占位符

```yaml
# 规则：花括号包裹，camelCase，语义清晰
✅ {scaleName}      # 量表名称
✅ {score}          # 总分
✅ {level}          # 等级标签

❌ {scale_name}     # 不用 snake_case
❌ {SCALE_NAME}     # 不用全大写
❌ {SN}             # 不用缩写
❌ {{scaleName}}    # 不用双花括号
```
