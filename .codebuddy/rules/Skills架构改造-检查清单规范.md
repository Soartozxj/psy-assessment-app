## ✅ 改造检查清单

### 6.1 代码审查清单（必须全部通过）

**每个插件改造完成后，必须检查以下项目**：

#### 架构符合性检查

- [ ] 插件继承了 `PluginBase` 类
- [ ] 插件使用了 `export default` 导出
- [ ] 插件放在正确的目录（`core/` 或 `optional/`）
- [ ] 插件文件名符合 `{插件名}-plugin.js` 格式
- [ ] 没有修改核心基类文件

#### 代码质量检查

- [ ] 所有异步函数都有 `try-catch` 异常处理
- [ ] 没有使用 `console.log()`（改用 `Adapter.logger`）
- [ ] 没有使用全局变量（所有状态封装在插件内部）
- [ ] 没有使用内联事件处理（改用 `EventHub` 或 `addEventListener`）
- [ ] 没有使用 `eval()` 或 `Function()` 构造函数

#### 安全性检查

- [ ] 所有用户输入都进行了验证和转义
- [ ] API密钥没有硬编码，存储在安全位置
- [ ] 没有XSS漏洞（正确使用 `textContent` 或转义）
- [ ] 没有SQL注入风险（使用参数化查询）

#### 性能检查

- [ ] 没有在循环中进行DOM操作
- [ ] 没有频繁触发重排/重绘
- [ ] 事件监听器在销毁时正确清理
- [ ] 没有内存泄漏风险

#### 兼容性检查

- [ ] 使用 `Adapter` 进行双端适配（没有硬编码H5或小程序的API）
- [ ] 保持了向后兼容（旧代码仍然可以正常工作）
- [ ] 所有原有功能都通过测试

### 6.2 测试验证清单（必须全部通过）

**每个插件必须通过以下测试**：

#### 单元测试（可选）

- [ ] 插件初始化测试：`init()` 方法正常执行
- [ ] 插件执行测试：`execute()` 方法正常返回结果
- [ ] 插件销毁测试：`destroy()` 方法正确清理资源
- [ ] 异常处理测试：错误情况被正确捕获和处理

#### 集成测试（必须）

- [ ] 插件加载测试：`PluginLoader.load('xxx')` 成功加载插件
- [ ] 插件获取测试：`PluginLoader.get('xxx')` 返回插件实例
- [ ] 事件通信测试：插件间通过 `EventHub` 正常通信
- [ ] 双端适配测试：在H5和小程序端都能正常运行

#### 手动测试（必须）

- [ ] 所有原有功能正常工作
- [ ] 新增功能正常工作
- [ ] 错误处理正常（显示友好的错误提示）
- [ ] 性能没有明显下降

## 📋 规则详解

### 代码审查清单详解

#### 为什么需要代码审查清单？

**原因**：

- 确保代码质量，避免低级错误
- 确保所有规则都被遵循
- 提供标准化的审查流程

#### 如何使用审查清单？

1. **自查**：开发人员在提交代码前，先对照清单自查
2. **互查**：团队成员交叉审查，对照清单逐项检查
3. **记录**：记录检查结果，对不符合项进行整改

#### 各检查项说明

##### 架构符合性检查

- **插件继承PluginBase**：确保所有插件都基于统一基类
- **使用export default**：确保模块正确导出
- **正确目录和文件名**：确保文件结构规范
- **不修改核心基类**：确保架构稳定性

##### 代码质量检查

- **异步函数try-catch**：确保异常被正确处理
- **不使用console.log**：确保使用统一的日志系统
- **不使用全局变量**：确保代码封装性
- **不使用内联事件处理**：确保事件解耦
- **不使用eval/Function**：确保安全性和性能

##### 安全性检查

- **输入验证和转义**：防止注入攻击
- **API密钥保护**：防止敏感信息泄露
- **XSS防护**：防止跨站脚本攻击
- **SQL注入防护**：防止数据库攻击

##### 性能检查

- **避免循环中DOM操作**：减少重排重绘
- **避免频繁重排重绘**：优化渲染性能
- **事件监听器清理**：防止内存泄漏
- **内存泄漏检查**：确保长期运行稳定性

##### 兼容性检查

- **双端适配**：确保在H5和小程序都能运行
- **向后兼容**：确保旧代码仍然可用
- **功能测试**：确保所有功能正常

### 测试验证清单详解

#### 为什么需要测试验证清单？

**原因**：

- 确保功能正确性
- 确保性能达标
- 确保兼容性

#### 如何使用测试验证清单？

1. **单元测试**：针对单个插件的功能测试
2. **集成测试**：针对插件间交互的测试
3. **手动测试**：针对用户界面的测试

#### 各测试项说明

##### 单元测试

- **初始化测试**：确保插件正确初始化
- **执行测试**：确保插件主要功能正常
- **销毁测试**：确保插件正确清理资源
- **异常处理测试**：确保错误情况被正确处理

##### 集成测试

- **插件加载测试**：确保插件能被正确加载
- **插件获取测试**：确保插件能被正确获取
- **事件通信测试**：确保插件间能正常通信
- **双端适配测试**：确保在不同平台都能运行

##### 手动测试

- **功能测试**：确保所有功能都能正常使用
- **错误处理测试**：确保错误信息友好
- **性能测试**：确保性能没有下降

## ✅ 检查清单使用流程

### 步骤1：开发完成

- 完成插件开发
- 完成代码自查

### 步骤2：代码审查

- 对照代码审查清单逐项检查
- 记录检查结果
- 对不符合项进行整改

### 步骤3：测试验证

- 运行单元测试
- 运行集成测试
- 进行手动测试

### 步骤4：提交审查

- 提交代码审查申请
- 提供检查清单结果
- 提供测试报告

### 步骤5：合并代码

- 审查通过后合并代码
- 部署到测试环境
- 进行回归测试
