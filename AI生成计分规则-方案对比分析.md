# AI 生成量表计分规则：方案对比分析文档

**文档版本：** 1.0
**创建日期：** 2026-05-05
**目的：** 对比分析 AI 生成量表计分规则的多种实现方案，为系统改造提供决策依据

---

## 一、背景与问题定义

### 1.1 当前流程（两步法）

```
手册原文 → [计分提取提示词] → 计分说明（自然语言） → [计分解析提示词 + 量表字段] → JSON
```

**存在问题：**

1. **Token 消耗高**：每次需 14,000+ tokens
2. **串联故障风险**：第1步失败导致第2步无法进行
3. **JSON 格式不稳定**：第2步可能输出非 JSON 内容

### 1.2 设计目标

| 目标               | 优先级 | 说明                        |
| ------------------ | ------ | --------------------------- |
| 降低 Token 消耗    | 高     | 节省 API 成本               |
| 保证 JSON 输出稳定 | 高     | 避免解析失败                |
| 保持准确度         | 高     | 计分规则不能出错            |
| 实现复杂度可控     | 中     | 平衡开发成本                |
| 支持长手册         | 中     | 部分量表手册超过 10k tokens |

## 三、方案详细说明

### 方案A：两步流程 + 验证重试

#### 3.1.1 实现流程

```
┌─────────────────────────────────────────────────────────┐
│ 第1步：计分提取                                        │
│ 输入：手册原文（可能很长）                              │
│ 输出：计分说明（自然语言）                               │
│ 提示词：prompts/计分提取.md                           │
└────────────────┬────────────────────────────────────────┘
                 │
                 ↓ 验证
        ┌────────────────┐
        │ 输出是否包含     │──否──→ 重试（最多3次）
        │ "计分规则"开头？ │
        └────────┬───────┘
                 │ 是
                 ↓
┌─────────────────────────────────────────────────────────┐
│ 第2步：计分解析                                        │
│ 输入：计分说明 + 量表字段信息                           │
│ 输出：JSON                                             │
│ 提示词：prompts/计分解析.md                           │
└────────────────┬────────────────────────────────────────┘
                 │
                 ↓ 验证
        ┌────────────────┐
        │ JSON.parse 是否 │──否──→ 重试（最多3次）
        │ 成功？          │
        └────────┬───────┘
                 │ 是
                 ↓
            ┌──────────┐
            │ 输出JSON  │
            └──────────┘
```

#### 3.1.2 代码示例

```javascript
async function generateScoringRules(manualText, scaleFields) {
  // 第1步：计分提取（带重试）
  let extractResult = null;
  for (let i = 0; i < 3; i++) {
    const result = await callAI(EXTRACT_PROMPT, manualText);
    if (result.content.includes('计分规则')) {
      extractResult = result;
      break;
    }
    console.log(`第1步重试 ${i + 1}/3`);
  }

  if (!extractResult) {
    throw new Error('计分提取失败（已重试3次）');
  }

  // 第2步：计分解析（带重试）
  let json = null;
  for (let i = 0; i < 3; i++) {
    const result = await callAI(
      PARSE_PROMPT,
      `计分规则文字：\n${extractResult.content}\n\n量表字段信息：\n${JSON.stringify(scaleFields)}`
    );

    try {
      json = JSON.parse(result.content);
      break;
    } catch (e) {
      console.log(`第2步重试 ${i + 1}/3`);
    }
  }

  if (!json) {
    throw new Error('计分解析失败（已重试3次）');
  }

  return json;
}
```

#### 3.1.3 优点与缺点

**✅ 优点：**

1. 改动小，快速上线
2. 准确度有保障（经过验证）
3. 容错能力强（重试机制）

**❌ 缺点：**

1. Token 消耗高（14k+）
2. 两次 API 调用，延迟高
3. 仍有可能两次都失败

#### 3.1.4 适用场景

- 快速上线 MVP
- 对成本不敏感的内部系统
- 需要最高准确度的临床量表

### 方案B：模板生成 JSON（半 AI）

#### 3.2.1 核心思路

**将"规则解析"从 AI 转移到代码模板**，只有"解释文字生成"用 AI。

```
手册原文 → [AI第1步：提取计分规则] → 计分说明（自然语言）
                                               ↓
                                               ↓ 代码解析（非AI）
                                               ↓
                                    [模板引擎：规则→JSON映射]
                                               ↓
                                               ↓ AI第2步：生成interpretation
                                               ↓
                                           完整JSON
```

#### 3.2.2 实现流程

```
┌─────────────────────────────────────────────────────────┐
│ 第1步：AI 提取计分规则                                │
│ 输入：手册原文                                        │
│ 输出：结构化计分说明（固定格式）                        │
│ 提示词：改进版 计分提取.md（要求结构化输出）            │
└────────────────┬────────────────────────────────────────┘
                 │
                 ↓
┌─────────────────────────────────────────────────────────┐
│ 第2步：模板引擎（代码）                                │
│ 输入：计分说明（结构化文本）                            │
│ 处理：正则表达式提取维度、指标、公式                    │
│ 输出：JSON 骨架（dimensions, metrics, screening）      │
└────────────────┬────────────────────────────────────────┘
                 │
                 ↓
┌─────────────────────────────────────────────────────────┐
│ 第3步：AI 生成 interpretation                          │
│ 输入：JSON 骨架 + 计分说明中的划界标准                 │
│ 输出：完整的 interpretation 数组                       │
│ 提示词：专用 interpretation 生成提示词                 │
└────────────────┬────────────────────────────────────────┘
                 │
                 ↓
            合并 → 完整JSON
```

#### 3.2.3 模板引擎伪代码

```javascript
// 从计分说明中提取维度定义
function extractDimensions(ruleText) {
  const dimensions = [];
  const dimensionPattern =
    /维度[：:]\s*(\w+)\s*[（(]([^)）]+)[）)]\s*题号[：:]\s*([\d,\-\s]+)\s*公式[：:]\s*(SUM|AVG)/g;

  let match;
  while ((match = dimensionPattern.exec(ruleText)) !== null) {
    dimensions.push({
      key: pinyin(match[1]), // 中文转拼音作为key
      label: match[1],
      items: match[3].trim(),
      formula: match[4]
    });
  }

  return dimensions;
}

// 生成 JSON 骨架
function generateJSONSkeleton(dimensions, metrics, screening) {
  return {
    dimensions: dimensions,
    metrics: metrics,
    interpretation: [], // 待AI填充
    screening: screening
  };
}

// 调用 AI 生成 interpretation
async function generateInterpretation(jsonSkeleton, ruleText) {
  const prompt = `你是心理测量学专家。根据以下计分规则和JSON骨架，生成interpretation数组。

计分规则：
${ruleText}

JSON骨架：
${JSON.stringify(jsonSkeleton, null, 2)}

要求：
1. 为每个 dimension 和 metric 生成分档解释
2. text 字段 30-80 字
3. 只输出纯JSON数组，不要markdown代码块`;

  const result = await callAI(prompt, '');
  return JSON.parse(result.content);
}
```

#### 3.2.4 优点与缺点

**✅ 优点：**

1. **Token 消耗降低 40-50%**（相比纯两步）
2. **JSON 结构稳定**（模板保证格式正确）
3. **容错能力强**（第2步是代码，不会失败）
4. **interpretation 质量高**（专门的 AI 调用）

**❌ 缺点：**

1. 需要实现模板引擎（工作量中等）
2. 计分说明必须结构化（对提示词有要求）
3. 如果手册格式异常，正则可能提取失败

#### 3.2.5 适用场景

- 平衡性能与稳定性
- 量产环境（需要高稳定性）
- 手册格式相对标准（如标准化量表）

### 方案C：分块处理长手册

#### 3.3.1 核心思路

**将长手册按章节分割，只处理计分相关章节。**

```
长手册原文 → [章节识别] → 计分相关章节 → [AI处理] → JSON
                ↓
            忽略其他章节
```

#### 3.3.2 实现流程

```
┌─────────────────────────────────────────────────────────┐
│ 第1步：章节识别与过滤                                  │
│ 输入：完整手册原文                                     │
│ 处理：正则表达式识别章节标题                             │
│ 输出：计分相关章节文本                                 │
│       （计分方法、维度定义、划界标准、解释说明）          │
└────────────────┬────────────────────────────────────────┘
                 │
                 ↓
┌─────────────────────────────────────────────────────────┐
│ 第2步：AI 生成 JSON                                   │
│ 输入：计分相关章节 + 量表字段                          │
│ 输出：JSON                                             │
│ 提示词：改进版 计分合并.md                            │
└────────────────┬────────────────────────────────────────┘
                 │
                 ↓ 验证
        ┌────────────────┐
        │ JSON.parse 是否 │──否──→ 将更多章节加入输入重试
        │ 成功？          │
        └────────┬───────┘
                 │ 是
                 ↓
            输出JSON
```

#### 3.3.3 章节识别伪代码

```javascript
function extractRelevantSections(manualText) {
  // 定义计分相关章节关键词
  const scoringKeywords = [
    '计分方法',
    '计分规则',
    '维度定义',
    '因子定义',
    '划界标准',
    '分数解释',
    '常模',
    '评分标准',
    '题目分配',
    '维度划分',
    '计分公式'
  ];

  // 按章节标题分割
  const sections = manualText.split(/(?=^#{1,3}\s+.+$)/m);

  // 过滤相关章节
  const relevantSections = sections.filter((section) => {
    return scoringKeywords.some((keyword) => section.includes(keyword));
  });

  // 如果未找到相关章节，返回全文的前5000字
  if (relevantSections.length === 0) {
    return manualText.substring(0, 5000);
  }

  return relevantSections.join('\n\n');
}
```

#### 3.3.4 优点与缺点

**✅ 优点：**

1. **Token 消耗最低**（只处理相关章节）
2. 支持超长手册（10k+ tokens）
3. 过滤无关内容，提高准确度

**❌ 缺点：**

1. 章节识别可能失败（手册格式不标准）
2. 可能遗漏跨章节的计分信息
3. 实现复杂度高

#### 3.3.5 适用场景

- 超长手册（如 SCL-90 完整手册）
- 手册格式标准（有清晰章节标题）
- 对 Token 消耗极度敏感

### 方案D：本地 + 云端混合

#### 3.4.1 核心思路

**用本地规则引擎处理确定性的部分，只把需要推理的部分发到云端 AI。**

```
手册原文 → [本地规则引擎] → 结构化信息
                ↓
            确定性部分（维度、指标、公式）
                ↓
         [云端AI] → interpretation 文字
                ↓
            合并 → 完整JSON
```

#### 3.4.2 实现流程

```
┌─────────────────────────────────────────────────────────┐
│ 第1步：本地规则引擎                                    │
│ 输入：手册原文                                        │
│ 处理：                                                │
│   - 正则表达式提取维度定义                             │
│   - 关键词匹配识别公式类型（SUM/AVG）                  │
│   - 表格解析提取划界分                                 │
│ 输出：部分结构化数据                                  │
└────────────────┬────────────────────────────────────────┘
                 │
                 ↓ 缺失？
        ┌────────────────┐
        │ 是否有缺失信息？ │──是──→ [云端AI补充]
        └────────┬───────┘
                 │ 否
                 ↓
┌─────────────────────────────────────────────────────────┐
│ 第2步：云端AI生成 interpretation                       │
│ 输入：结构化数据 + 原始手册（节选）                    │
│ 输出：interpretation 数组                             │
│ Token消耗：~2000                                      │
└────────────────┬────────────────────────────────────────┘
                 │
                 ↓
            合并 → 完整JSON
```

#### 3.4.3 本地规则引擎伪代码

```javascript
function localRuleEngine(manualText) {
  const result = {
    dimensions: [],
    metrics: [],
    interpretation: [],
    screening: null,
    missing: []
  };

  // 1. 提取维度定义（正则）
  const dimPattern = /(\w+)\s*[（(]?\s*包含题目[）)]?\s*[：:]\s*([\d,\-\s]+)/g;
  let match;
  while ((match = dimPattern.exec(manualText)) !== null) {
    result.dimensions.push({
      key: pinyin(match[1]),
      label: match[1],
      items: match[2].trim(),
      formula: inferFormula(manualText, match[1]) // 推理公式
    });
  }

  // 2. 如果未找到维度，标记为缺失
  if (result.dimensions.length === 0) {
    result.missing.push('dimensions');
  }

  // 3. 提取划界分（正则）
  const cutoffPattern = /(\d+)\s*[-~]\s*(\d+)\s*[:：]\s*(\w+)/g;
  // ... 解析划界分

  return result;
}

// 推理公式类型
function inferFormula(text, dimensionName) {
  if (text.includes('求和') || text.includes('总分') || text.includes('相加')) {
    return 'SUM';
  }
  if (text.includes('平均') || text.includes('均分')) {
    return 'AVG';
  }
  return 'SUM'; // 默认SUM
}
```

#### 3.4.4 优点与缺点

**✅ 优点：**

1. **Token 消耗最低**（云端部分仅 ~2000）
2. **响应速度快**（本地处理 + 小模型调用）
3. **成本低**（本地免费）

**❌ 缺点：**

1. **实现复杂度最高**（需要开发本地规则引擎）
2. 准确度依赖规则完整性
3. 维护成本高（需要持续更新规则）

#### 3.4.5 适用场景

- 高性能要求（低延迟）
- 大批量处理（如批量导入100+量表）
- 有开发资源投入

### 方案E：一步合并

#### 3.5.1 核心思路

**将两步合并为一步，直接从手册原文生成 JSON。**

```
手册原文 + 量表字段 → [合并提示词] → JSON
```

#### 3.5.2 实现流程

```
┌─────────────────────────────────────────────────────────┐
│ 一步调用 AI                                            │
│ 输入：手册原文 + 量表字段信息 + 合并提示词              │
│ 输出：JSON                                             │
│ 提示词：prompts/计分合并.md                           │
└────────────────┬────────────────────────────────────────┘
                 │
                 ↓ 验证
        ┌────────────────┐
        │ JSON.parse 是否 │──否──→ 重试（最多3次）
        │ 成功？          │
        └────────┬───────┘
                 │ 是
                 ↓
            输出JSON
```

#### 3.5.3 优点与缺点

**✅ 优点：**

1. **Token 消耗最低**（~3.5k）
2. **实现最简单**（只需一个提示词）
3. **延迟最低**（一次 API 调用）

**❌ 缺点：**

1. **JSON 格式不稳定**（长上下文容易输出多余文字）
2. **准确度未验证**（需要更多测试）
3. 调试困难（无法看到中间结果）

#### 3.5.4 适用场景

- 简单量表（如 PHQ-9 只有9题）
- 快速验证原型
- 对成本极度敏感

## 四、推荐方案与实施建议

### 4.1 推荐方案：方案B（模板生成 JSON）

**推荐理由：**

1. **平衡性能与稳定性**（Token 降低 40-50%，JSON 格式稳定）
2. **容错能力强**（模板保证结构，AI 只生成文本）
3. **实现复杂度可控**（中等工作量）

### 4.2 实施路线图

```
Phase 1（1-2天）：方案A实施
  - 实现两步+验证重试
  - 上线验证基本功能

Phase 2（3-5天）：方案B实施
  - 开发模板引擎
  - 改进提示词（要求结构化输出）
  - A/B 测试对比

Phase 3（可选）：性能优化
  - 如果 Token 消耗仍高，考虑方案C或D
  - 针对超长手册优化
```

### 4.3 风险控制

| 风险                        | 应对措施                |
| --------------------------- | ----------------------- |
| 模板引擎解析失败            | 回退到方案A（两步流程） |
| AI 生成 interpretation 失败 | 使用默认解释文字        |
| 量表类型判断错误            | 增加人工审核环节        |
